## Context

### Problem Statement

Canon-services needs to integrate with multiple external service providers:

- LLM providers (Anthropic, OpenAI, Groq) for AI capabilities
- Storage providers (AWS S3) for file storage
- Communication providers (Twilio, Resend) for notifications
- Market data providers (Perplexity, Exa, H1B) for research

Previously, endpoint classes were **centralized** in `vendor/providers/`, causing:

1. **Duplication**: `embed/endpoints/openai.py` duplicated by `vendor/providers/embed/`
2. **Cluttered vendor folder**: All providers mixed together regardless of domain
3. **Import complexity**: VendorService needed to know about all endpoint locations

**Why This Matters**: Code duplication leads to divergent behavior, maintenance burden, and
confusion about which implementation is canonical.

### Background

**Current State** (before this decision):

- Endpoints scattered across `embed/`, `market/`, and `vendor/providers/`
- Duplicate implementations existed in multiple locations
- `vendor/match.py` had hardcoded provider matching logic
- No unified way to discover all available endpoints

**Driving Forces**:

- **Domain cohesion**: Endpoints should live near their service logic
- **Single source of truth**: No duplicate implementations
- **Extensibility**: Easy to add new providers without modifying central code
- **Kron compatibility**: LLM providers should use proven kron code

### Assumptions

1. Endpoint classes are imported at module load time (decorator works at import)
2. Kron remains the standard for LLM endpoint implementations
3. Provider/endpoint keys are stable and case-insensitive

### Constraints

| Type      | Constraint                                  | Impact                                                |
| --------- | ------------------------------------------- | ----------------------------------------------------- |
| Technical | Must use kron's `Endpoint` base class       | Endpoint implementations must inherit from kron       |
| Technical | Registration happens at import time         | Services must import their endpoints in `__init__.py` |
| Business  | Must not break existing VendorService usage | `match_endpoint()` API stays compatible               |

## Decision

### Summary

**We will** adopt a decorator-based endpoint registry that allows endpoints to be scattered across
service folders while providing a unified resolution mechanism with automatic kron fallback.

### Rationale

**Key factors in the decision**:

1. **Domain cohesion over centralization**: Endpoint logic belongs with its service (e.g.,
   `embed/endpoints/` for embedding endpoints)
2. **Decorator pattern familiarity**: Python developers expect decorator-based registration (like
   `@app.route()`)
3. **Kron as default**: LLM providers have mature implementations in kron; don't reinvent
4. **Single resolution function**: `match_endpoint()` works for all providers regardless of
   implementation location

### Implementation Approach

```python
# 1. Register endpoint with decorator (in feature or service folder)
from canon.utils.endpoints import register_endpoint

@register_endpoint(provider="aws", endpoint="s3")
class S3Endpoint(Endpoint):
    """S3 storage endpoint."""
    ...

# 2. Resolve endpoint from anywhere
from canon.utils.endpoints import match_endpoint

# Custom registered endpoint
s3 = match_endpoint("aws", "s3", bucket="my-bucket")

# Kron native (automatic fallback)
claude = match_endpoint("anthropic", "messages")
```

**Resolution Order**:

1. Check canon-core `_ENDPOINT_REGISTRY` for custom endpoint
2. Fall back to `canon.services.vendor.match()` for LLM providers
3. Kron handles OpenAI-compatible fallback for unknown providers

### Alternatives Considered

#### Alternative 1: Centralized Provider Dictionary

**Description**: Keep all endpoint classes in a single dictionary in `vendor/providers/`

| Criterion       | Score (1-5) | Notes                           |
| --------------- | ----------- | ------------------------------- |
| Domain cohesion | 2           | All endpoints mixed together    |
| Extensibility   | 2           | Requires modifying central file |
| Familiarity     | 4           | Simple dictionary lookup        |

**Why Not Chosen**: Violates separation of concerns; endpoint logic should live with service logic

#### Alternative 2: Plugin Discovery System

**Description**: Use `importlib` entry points or directory scanning for auto-discovery

| Criterion       | Score (1-5) | Notes                            |
| --------------- | ----------- | -------------------------------- |
| Domain cohesion | 5           | Endpoints in service folders     |
| Extensibility   | 5           | Zero-config addition             |
| Complexity      | 2           | Entry points or runtime scanning |
| Debuggability   | 2           | Hard to trace what's registered  |

**Why Not Chosen**: Over-engineered for current scale; explicit imports are clearer

#### Alternative 3: Abstract Factory Pattern

**Description**: Create factory classes for each provider category

| Criterion   | Score (1-5) | Notes                         |
| ----------- | ----------- | ----------------------------- |
| Type safety | 5           | Factory enforces return types |
| Boilerplate | 2           | Lots of factory classes       |
| Familiarity | 3           | Less Pythonic                 |

**Why Not Chosen**: Too much ceremony; decorator pattern is simpler and sufficient

### Decision Matrix

| Criterion          | Weight | Centralized | Plugin  | Factory | **Decorator** |
| ------------------ | ------ | ----------- | ------- | ------- | ------------- |
| Domain cohesion    | 25%    | 2           | 5       | 4       | **5**         |
| Extensibility      | 25%    | 2           | 5       | 4       | **4**         |
| Simplicity         | 20%    | 4           | 2       | 2       | **5**         |
| Debuggability      | 15%    | 4           | 2       | 3       | **4**         |
| Familiarity        | 15%    | 4           | 3       | 3       | **5**         |
| **Weighted Total** | 100%   | **2.8**     | **3.6** | **3.3** | **4.6**       |

## Consequences

### Positive Consequences

1. **Endpoints live with services**: `embed/endpoints/`, `market/endpoints/` contain relevant code
2. **No duplication**: Removed `vendor/providers/embed/`, `vendor/providers/market/perplexity/`,
   `vendor/providers/market/exa/`
3. **Kron integration**: LLM providers use battle-tested kron implementations
4. **Simple API**: Single `match_endpoint()` function for all providers
5. **Easy extension**: Add `@register_endpoint` decorator to new endpoint class

### Negative Consequences

1. **Import order dependency**: Service `__init__.py` must import endpoint modules for registration
   - _Mitigation_: Document import requirements in each service

2. **Global state**: `_ENDPOINT_REGISTRY` is module-level singleton
   - _Mitigation_: Provide `EndpointRegistry.clear()` for testing

3. **Silent kron fallback**: Unknown providers don't error, they fall through to kron
   - _Mitigation_: Kron logs warning for unknown providers; `is_supported_provider()` for
     explicit checks

### Neutral Consequences

1. **Two resolution paths**: Custom endpoints vs kron native providers handled differently
   internally (but same API)

### Risks

| Risk                        | Likelihood | Impact | Mitigation                                                 |
| --------------------------- | ---------- | ------ | ---------------------------------------------------------- |
| Registration key collisions | Low        | Medium | `ValueError` on duplicate, `override=True` for intentional |
| Import side effects         | Low        | Low    | Registration is idempotent (same class)                    |
| Kron version drift          | Medium     | Low    | Pin kron version in dependencies                           |

### Dependencies Introduced

| Dependency | Type    | Version    | Stability | Notes                              |
| ---------- | ------- | ---------- | --------- | ---------------------------------- |
| kron       | Library | >= 1.0.0   | Stable    | Required for LLM endpoint fallback |

### Migration Impact

**Backwards Compatibility**: Compatible

The `match_endpoint()` function in `canon.utils.endpoints` provides the same API as the
previous `vendor.match.match_endpoint()`. The `vendor.match` module now imports from
`canon.utils.endpoints` for compatibility.

**Migration Steps**:

1. Update imports: `from canon.utils.endpoints import match_endpoint`
2. Remove duplicate endpoint files (already done)
3. Ensure service/feature `__init__.py` imports endpoint modules
